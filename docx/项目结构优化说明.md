# 项目结构优化说明

## 目标

本次结构优化的目标是把编辑器相关能力从“单文件堆叠实现”演进为“分层、配置化、可持续扩展”的架构，减少耦合，降低维护成本，并为后续新增功能保留清晰边界。

## 设计原则

- 单一职责：一个模块只负责一类问题。
- 分层解耦：状态、动作、视图、静态配置分开。
- 配置驱动：菜单、快捷键、文案尽量由配置提供。
- 保持兼容：外部调用方式尽量不变，避免改动扩散。
- 可持续演进：新功能优先新增模块，不优先扩写大文件。

## 当前架构

### 1. 应用编排层

路径：`src/App.tsx`

职责：
- 组织编辑器和工具栏

说明：
- `App.tsx` 现在只承担页面级编排。
- 工作区状态已经抽到 `src/hooks/useEditorWorkspace.ts`。

### 2. 工作区状态层
职责：
- 管理 `content`、`theme`、`currentFilePath`、`editorKey`

说明：
- 新建文件会自动清空当前路径，避免保存污染。
- 打开文件时统一重建编辑器实例，减少历史状态残留。
职责：
- 初始化 Milkdown 编辑器
- 暴露编辑器命令接口
- 处理主题样式和编辑区挂载
- 组合自定义数学块和语言选择器

拆分子模块：
- `src/components/milkdown/types.ts`
- `src/components/milkdown/mathBlock.ts`
- `src/components/milkdown/useCodeBlockLanguageSelector.ts`
- `src/components/milkdown/CodeBlockLanguageSelector.tsx`
- `src/components/milkdown/codeBlockLanguages.ts`

### 4. 工具栏控制层

路径：`src/components/toolbar/useToolbarController.ts`

职责：
- 文件打开、保存、另存为
- 编辑动作触发
- 快捷键监听
- 工具栏 UI 状态管理

说明：
- 控制层只处理“做什么”，不直接承担 UI 渲染。
- 状态与动作被显式拆分为 `ToolbarUiState` 和 `ToolbarActions`。

### 5. 工具栏视图层

路径：
- `src/components/toolbar/ToolbarMenus.tsx`
- `src/components/toolbar/ToolbarDialogs.tsx`
- `src/components/toolbar/toolbarMenuConfig.ts`

职责：
- `ToolbarMenus.tsx` 只负责菜单展示和事件绑定
- `ToolbarDialogs.tsx` 只负责对话框展示和表单交互
- `toolbarMenuConfig.ts` 负责描述菜单结构、分组、快捷键提示和动作映射

### 6. 静态配置层

路径：`src/components/toolbar/toolbarConfig.ts`

职责：
- 维护关于页文案
- 维护快捷键分组数据
- 维护与 UI 呈现无关的静态说明信息

说明：
- 静态内容不再散落在 JSX 中。
- 后续新增快捷键、调整分组、修改文案时，只需要改配置层。
- 菜单本身也已经开始配置化，视图层只保留最薄的渲染逻辑。

## 关键收益

- 降低单文件复杂度
- 提升模块复用性
- 新增菜单项时不必修改大段布局代码
- 新增编辑器能力时，边界更清楚
- 更容易做测试、替换和后续扩展

## 后续建议

- 将工具栏菜单进一步数据化，统一生成菜单 UI。
- 为 Toolbar 动作增加更明确的 action map 层，减少散点函数。
- 把工作区状态进一步抽象为 `AppModel` 或 `workspace store`，方便未来接入持久化、历史记录、最近文件等能力。
- 为关键动作补充单元测试或轻量集成测试。

## 结论

当前结构已经从“功能堆叠”转向“可持续演进”的分层架构。后续优化应优先沿着“配置驱动、动作中心化、视图薄化”三个方向推进。
