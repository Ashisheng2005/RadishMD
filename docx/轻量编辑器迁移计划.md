# 轻量编辑器迁移计划

## 目标

- 统一以 Markdown 源文本作为唯一事实来源。
- 去掉 Milkdown / ProseMirror 中复杂的 node view 依赖。
- 把公式块、代码块、表格等能力收敛成轻量、可维护的块组件。
- 保留现有 Tauri、React、工具栏、文件读写、主题系统。

## 核心原则

- 文档层只保存一份 Markdown 字符串。
- 编辑层只负责局部交互，不维护复杂文档树。
- 块级交互由独立组件处理，不依赖重型 schema / selection 生态。
- 编辑态与预览态分离，但都围绕同一份 Markdown 数据。

## 推荐架构

### 1. DocumentStore

职责：
- 管理当前 Markdown 文本。
- 提供 `load`、`save`、`insertAt`、`deleteRange`、`replaceRange` 等操作。

要求：
- 只有这一层能写入最终文档内容。
- 工具栏、快捷键、块编辑都通过它更新内容。

### 2. BlockParser

职责：
- 把 Markdown 解析成块列表。
- 识别段落、标题、公式块、代码块、列表、引用、表格等。

建议：
- 第一版只支持常用块。
- 复杂语法后补，不一开始追求完整覆盖。

### 3. BlockRenderer

职责：
- 根据块类型渲染不同组件。
- 普通文本、公式块、代码块、表格都用各自组件处理。

要求：
- 不再依赖 node view 承载全部交互。
- 每个块只处理自己的局部状态。

### 4. SelectionController

职责：
- 管理当前焦点块、块内光标位置、选区范围。
- 处理块间切换和焦点恢复。

说明：
- 这是轻量编辑器的核心协调层。
- 作用是替代 ProseMirror 的复杂 selection 体系。

### 5. ShortcutController

职责：
- 统一处理键盘快捷键。
- 例如：Enter、Shift+Enter、Escape、Backspace、Ctrl/Command 组合键。

要求：
- 快捷键逻辑与数据层分离。
- 工具栏和键盘行为共用同一套命令。

### 6. FormulaBlock

职责：
- 负责数学公式块的预览和编辑。
- 预览用 KaTeX。
- 编辑用 textarea。

交互规则：
- 点击进入编辑。
- 失焦自动保存。
- Shift+Enter / Ctrl+Enter 完成编辑。
- Escape 取消。
- 空内容时 Backspace 删除整个块。

### 7. Serializer

职责：
- 块状态转 Markdown。
- Markdown 转块状态。

要求：
- 保证保存与重新打开一致。
- 作为块系统与文档字符串之间的唯一转换层。

## 迁移顺序

### 阶段 1：统一数据源

目标：
- 让整个编辑器只围绕 Markdown 字符串工作。

要做的事：
- 打通打开、保存、另存为。
- 工具栏命令最终回写 Markdown 字符串。
- 先不换 UI，只先统一数据流。

### 阶段 2：实现 BlockParser

目标：
- 把 Markdown 文本切成块。

优先支持：
- 段落
- 标题
- 代码块
- 公式块

### 阶段 3：实现 FormulaBlock

目标：
- 用轻量组件替换复杂 node view。

优先行为：
- 点击编辑
- 失焦保存
- 删除空块
- KaTeX 预览

### 阶段 4：接管工具栏命令

目标：
- 让插入标题、插入代码块、插入公式块等命令接入新内核。

要求：
- 现有工具栏 UI 尽量不改。
- 先保证命令可用，再优化体验。

### 阶段 5：替换正文编辑区

目标：
- 用轻量编辑器视图替换 Milkdown 正文区。

范围：
- 先支持段落、标题、公式块、代码块。
- 再逐步扩展表格、列表、引用。

### 阶段 6：清理旧依赖

目标：
- 移除 Milkdown / ProseMirror 的重型依赖。

清理内容：
- node view
- schema 相关复杂逻辑
- selection 依赖
- 不再使用的编辑器上下文

## 最小可用版本范围

建议第一版先支持：
- 文本输入与保存
- 标题
- 段落
- 公式块
- 代码块
- 基本快捷键
- 文件打开 / 保存
- 主题切换

第一版暂不做：
- 复杂表格编辑
- 多光标
- 深层嵌套块交互
- 全量 Markdown 富文本能力

## 主要风险

- 编辑状态管理需要自己维护，不能完全依赖现成框架。
- 初期能力会比 Milkdown 少，但会更稳定、更可控。
- 迁移必须分阶段进行，不能一次性全换。

## 建议节奏

- 第 1-2 天：统一数据源，落地 DocumentStore。
- 第 3-4 天：完成 BlockParser 和 FormulaBlock。
- 第 5 天：接入工具栏命令。
- 第 6 天以后：逐步替换正文编辑区。

## 结论

如果目标是“简洁、高效、可维护”，建议统一到轻量编辑器方案，不再继续在复杂 node view 上投入。
